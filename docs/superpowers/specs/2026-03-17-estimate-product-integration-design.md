# Estimate Product Integration & Print Enhancement

**Date:** 2026-03-17
**Status:** Approved

## Overview

Three enhancements to the **admin estimate system only** (`estimates` + `estimate_items` tables):
1. Better product discovery with global search bar
2. Professional HTML-based print/PDF system (single renderer)
3. Column toggle controls with 2 new columns (Product Image, Pack Size)

**Scope:** Admin/staff estimates only. Painter estimates (`painter_estimates` table, `painter-estimate-create.html`) are a separate system and are **out of scope** for this work. Customer mode on this page is also out of scope for now — to be designed separately since customer OTP auth needs its own spec.

## 1. Product Search Enhancement

### Current State
- `estimate-create-new.html` loads 623 active products from `GET /api/products`
- Selection requires 3 steps: Brand → Category → Product (cascade dropdowns)
- Products stored in `allProducts` array client-side

### Changes

**Add a global search bar** above the cascade section:
- Text input with search icon, placeholder: "Search products by name..."
- Searches `allProducts` array client-side (no new API needed)
- Shows results in a dropdown panel below the search bar (max 10 results)
- Each result shows: product name, brand name, category name
- Clicking a result selects the product directly (skips cascade), populates `selectedProduct`, shows pack size / area inputs
- Cascade dropdowns remain below as alternate browse flow
- Clicking outside the results panel closes it

**Implementation:**
- New `<div>` with search input placed above the Brand/Category/Product row in Step 2
- `filterProducts(query)` function: case-insensitive name match on `allProducts`
- Results panel: positioned absolute, z-index above other content
- On select: calls existing `onProductSelected(value, data)` flow

## 2. Print System Redesign

### Current State
- Two separate renderers maintained independently:
  - PDFKit server-side (`routes/estimate-pdf-generator.js`, 223 lines of coordinate layout)
  - Browser print CSS (`estimate-view.html` with `@media print`)
- Design changes require updating both systems

### Changes

**New file: `public/estimate-print.html`**

A standalone HTML page that renders an estimate beautifully for both screen preview and printing. Single source of truth for estimate presentation.

**Access patterns:**
- From `estimate-view.html`: "Print" button opens `estimate-print.html?id={estimateId}`
- From `estimate-create-new.html`: After saving, option to open print preview
- PDF download: Server renders this same page via Puppeteer

**Page layout (A4 optimized):**

```
+---------------------------------------------+
|  [Logo]  QUALITY COLOURS                    |
|          Address | Phone | Email | GST      |
|                              ESTIMATE       |
|                              #EST20260317.. |

|---------------------------------------------|
|  BILL TO:                                   |

|  Phone | Address                            |
|---------------------------------------------|
|  [Column Toggle Pills - screen only]        |
|                                             |

|  | # | Image | Product | Pack | Qty |..|   |

```

**GST display rule:**
- The `estimates` table has a `show_gst_breakdown` flag
- When `show_gst_breakdown = true`: show Subtotal + GST line + Grand Total
- When `show_gst_breakdown = false` (default for most estimates): show only Grand Total with note "All prices inclusive of GST"
- The print page reads this flag from the estimate record

**Design:**
- Admin brand colors: primary `#667eea`, gradient to `#764ba2`
- Clean typography: system fonts, proper hierarchy
- Table: header with primary color background, alternating row stripes
- Product images: small thumbnails (40x40) when Image column enabled
- Responsive: A4 on desktop/print, mobile-friendly on screen
- `@media print` CSS: hides toggle pills, action buttons; forces white background

**Action bar (screen only, hidden on print):**
- Column toggle pills
- "Print" button (`window.print()`)
- "Download PDF" button
- "Back" button

**PDF endpoint change:**
- `GET /api/estimates/:id/pdf` — updated to use Puppeteer
- Requires: `npm install puppeteer-core` (reuses Chromium binary from whatsapp-web.js's bundled puppeteer)
- Chromium path discovery: use `require('puppeteer').executablePath()` to find the Chromium binary installed by the `puppeteer` package (whatsapp-web.js dependency). If unavailable, fall back to `/usr/bin/chromium-browser` or `/usr/bin/google-chrome`.
- `routes/estimate-pdf.js`: launches `puppeteer-core` with explicit `executablePath`, navigates to `estimate-print.html?id={id}&mode=pdf&token={token}`, generates PDF
- `mode=pdf` query param hides the action bar via CSS
- **Token auth flow for Puppeteer:** The server passes the user's session token as `?token=X` in the URL. The `estimate-print.html` page JS checks for `token` URL param first — if present, uses it as Bearer token for all API calls instead of localStorage `auth_token`. This allows Puppeteer (which has no localStorage) to authenticate.
- Returns PDF buffer as response with `Content-Disposition: attachment`

**Files affected:**
- `routes/estimate-pdf-generator.js` — kept as-is but no longer called from `estimate-pdf.js`. Can be removed later.
- `routes/estimate-pdf.js` — rewritten to use Puppeteer instead of calling `generateEstimatePDF()`

**Note:** Painter estimates have their own separate PDF generator (`routes/painter-estimate-pdf-generator.js`) which is NOT affected by this change.

## 3. Column Toggles

### Current State
- `estimates.column_visibility` JSON field exists in DB (used by POST/PUT in server.js)
- `estimate-view.html` has toggles for 5 columns: qty, mix, price, color, total
- `estimate-pdf-generator.js` reads colVis with 6 keys (adds breakdown)

### Changes

**8 toggleable columns:**

| Key | Label | Default | Notes |
|-----|-------|---------|-------|
| `show_image` | Image | ON | **NEW** — product thumbnail |
| `show_pack_size` | Pack Size | ON | **NEW** — e.g., "20L", "10KG" |
| `show_qty` | Qty/Area | ON | Existing |
| `show_mix` | Mix Info | ON | Existing |
| `show_price` | Unit Price | ON | Existing |
| `show_breakdown` | Breakdown | OFF | Existing (in DB, not in view UI) |
| `show_color` | Color Cost | OFF | Existing |
| `show_total` | Total | ON | Existing |

**Toggle UI (on both estimate-create-new.html and estimate-print.html):**
- Row of pill buttons above the items table
- Each pill: clickable, shows column name
- Active state: solid primary color with white text
- Inactive state: outline/gray border
- Clicking toggles column visibility instantly (no page reload)

**Persistence:**
- `localStorage` key: `estimate_column_prefs` — saves user's default toggle preferences
- On estimate save: current toggle state stored in `estimates.column_visibility` JSON
- On estimate view/print: reads from saved `column_visibility`, falls back to localStorage defaults

### Data changes for new columns

**Product Image column:**
- Source: `GET /api/products` already returns `image_url` per product (via pack_sizes → zoho_items_map JOIN)
- When adding item to estimate: store product's `image_url` on the estimate item
- DB: add `image_url VARCHAR(500)` column to `estimate_items` table
- Displayed as 40x40 thumbnail in table and print
- Archival: image URL is snapshotted at estimate creation time (product image may change later)
- Fallback: when `image_url` is NULL, show a gray placeholder icon (paint bucket SVG)

**Pack Size column:**
- Already available in the item data during creation (from `mix_info` field, e.g., "20L pack")
- Rather than adding a new DB column, **parse from existing `mix_info` at render time**
- Regex (case-insensitive): `\d+(\.\d+)?\s*(L|LTR?|Ltr|Litre|KG|Kg|M|PC)\b` — handles common unit variations
- This avoids data duplication (YAGNI — pack size is already in mix_info)

## 4. Migration

**New migration file: `migrations/migrate-estimate-columns.js`**

```sql

```

Only one column added. Pack size is parsed from existing `mix_info` at render time.

### [PAGE-165] DELETE customer-type sent without auth headers → likely 401
- **cat:** correctness · **phase:** 1 · **surface:** `public/admin-customer-types.html`
- deleteType() calls fetch(url, { method: 'DELETE' }) with NO headers, omitting the Bearer token that every other call on this page sends via getAuthHeaders(). On an auth-gated admin route this returns 401 and the delete silently fails ('Failed to delete'). Dead primary action.
- **evidence:** `admin-customer-types.html:268-270 fetch(`/api/customer-types/${id}`, { method: 'DELETE' }) — compare saveType which passes headers: getAuthHeaders()`

### [PAGE-171] Customer/project/notes data rendered into innerHTML with zero escaping (stored XSS from public form)
- **cat:** correctness · **phase:** 1 · **surface:** `public/admin-estimate-requests.html`
- This page consumes a PUBLIC form (request-estimate.html). loadRequests() and viewRequest() interpolate request.customer_name, phone, location, additional_notes, products_json item fields (product_name, brand_name, details) directly into innerHTML with no esc() — and no escape helper is even defined. An attacker submitting the public estimate form can inject HTML/script that runs in the admin panel.
- **evidence:** `admin-estimate-requests.html:326 `${request.customer_name}`, :450 `${req.additional_notes}`, :481-488 `${item.product_name}` / `${item.details}` from JSON.parse(req.products_json)`

### [PAGE-034] Customer page gated by STAFF auth — dead for its intended actor
- **cat:** correctness · **phase:** 1 · **surface:** `public/customer-requests.html`
- Page loads /js/auth-helper.js then calls checkAuthOrRedirect(), which checks isAuthenticated() = localStorage 'auth_token' (staff token). A real customer only has 'customer_token', so checkAuthOrRedirect() bounces them to /login.html (staff login). The customer 'My Requests' flow is broken end-to-end.
- **evidence:** `public/customer-requests.html:13-14; auth-helper.js:46-47 isAuthenticated reads auth_token`

### [PAGE-035] Calls staff-only requireAuth endpoints with no auth header + hardcoded phone fallback (IDOR/leak)

### [PAGE-301] Pending/rejected engineers are bounced to login — dashboard never renders for non-approved accounts
- **cat:** correctness · **phase:** 1 · **surface:** `public/engineer-dashboard.html`
- loadProjects() calls GET /me/projects, which is gated by requireEngineerAuth (approved-only): it returns 403 for status pending/suspended/rejected (routes/engineers.js:43-44). The page passes that response to EP.handleAuthFail(r), which treats 401/403 identically: it clears the session and redirects to /engineer-login.html (engineer-portal.js:25-32). So a freshly-registered pending engineer who signs in (login allows pending through to dashboard, login.html:176-181) lands on the dashboard, the /me/projects call 403s, the session is wiped, and they are kicked back to login in a loop — they can never actually use the portal until an admin approves them. The status banner ('pending approval') is therefore unreachable. This is a dead primary flow for every brand-new account.
- **evidence:** `engineer-dashboard.html:249-250 fetch /me/projects + handleAuthFail; engineer-portal.js:25-32 handleAuthFail treats 403 as logout; routes/engineers.js:43-44 requireEngineerAuth 403 when status!=='approved' & :790 /me/projects uses requireEngineerAuth`
